# System Architecture

Mind Measure is a cloud-native mental health monitoring platform serving two product lines — **University** (student wellbeing) and **Pro** (corporate employee wellbeing) — sharing common AWS infrastructure.

**Last Updated**: 28 February 2026

## Authentication

### AWS Cognito (Students and University/Company Users)

- **Region**: `eu-west-2` (London)
- **User Pool**: Separate pools for University and Pro
- **Features**: Email verification, password policies, JWT tokens

```typescript
const { signIn } = useAuth();
const result = await signIn(email, password);

const session = await fetchAuthSession();
const token = session.tokens?.accessToken?.toString();
```

### Superuser Authentication (Custom)

Passwordless 6-digit code flow for Mind Measure staff:

1. User enters `@mindmeasure.co.uk` email
2. Server generates 6-digit code, sends via AWS SES
3. User enters code, receives JWT token
4. Token stored in session, validated on each request

## API Layer

All API functions run as **Vercel serverless functions** in the `/api/` directory of each admin portal repo. There is no separate API Gateway or Lambda deployment.

### Core Endpoints


## Data Layer

### Amazon Aurora Serverless v2

- **Engine**: PostgreSQL
- **Region**: `eu-west-2` (London)
- **Scaling**: 0.5–16 ACUs (automatic)
- **Encryption**: AES-256 at rest, TLS in transit

**Connection pattern** (serverless functions):

```typescript
import { Client } from 'pg';

const client = new Client(getSecureDbConfig());
try {
  await client.connect();
  const result = await client.query(sql, params);
  return result.rows;
} finally {
  await client.end();
}
```

Use `Client` (not `Pool`) in serverless functions. Always call `client.end()` in a `finally` block. Never hardcode credentials — use `getSecureDbConfig()` from `api/_lib/db-config.ts`.

### Key Tables

| Table | Purpose |
|---|---|
| `core_users` | Mind Measure staff with `user_permission[]` enum |
| `university_authorized_users` | University admin accounts with invite flow |
| `profiles` | Student/employee profiles (linked to Cognito) |
| `universities` | University config, branding, settings |
| `companies` | Company config, branding, org structure |
| `assessment_sessions` | Check-in session metadata |
| `fusion_outputs` | Multi-modal analysis results |

## AI/ML Services

### AWS Bedrock (Claude 3.7 Sonnet)

- **Model**: `anthropic.claude-3-7-sonnet-20250219-v1:0`
- **Purpose**: Transcript analysis after each check-in
- **Endpoint**: `/api/bedrock/analyze-text`
- **Outputs**: Mood score, sentiment, conversation summary, positive/negative drivers, risk flags, custom question responses

### ElevenLabs Conversational AI (Jodie)

The **Jodie** voice agent conducts check-in conversations with users.

- **SDK**: `@elevenlabs/react` (`useConversation` hook)
- **Separate agents**: University and Pro each have their own ElevenLabs agent
- **Prompts**: Managed locally as `jodie_prompt_v2_uni.md` and `jodie_prompt_v2_pro.md`
- **Features**: Real-time voice conversation, transcript generation, dynamic variables (user name, last summary, check-in count)
- **Overrides**: First message, voice stability, voice similarity (must be enabled in ElevenLabs agent settings)

### AWS Rekognition

- **Purpose**: Facial emotion analysis during check-ins
- **Integration**: Video frames captured client-side, sent to `/api/rekognition/analyze-frame`
- **Features**: Emotion detection, face analysis

### Multi-Modal Analysis Pipeline

```
Student Check-in → ElevenLabs Jodie → Conversation + Transcript
                                            ↓
                                    Session Complete
                                            ↓
                    ┌───────────────────────┼───────────────────────┐
                    │                       │                       │
              Audio Analysis         Visual Analysis         Text Analysis
              (Client-side           (Rekognition)           (Bedrock
               Web Audio API)                                Claude 3.5)
                    │                       │                       │
                    └───────────────────────┼───────────────────────┘
                                            ↓
                                     V2 Fusion Engine
                                   70% text / 15% audio / 15% visual
                                            ↓
                                    Aurora Database
                                    (fusion_outputs)
```

## Security Architecture

### Network Security

- **TLS 1.3**: All communications
- **CORS**: Configured per domain, including `capacitor://localhost` for mobile
- **WAF**: Rate limiting, SQL injection prevention

### Data Security

- **Encryption at rest**: AES-256 (Aurora, S3)
- **Encryption in transit**: TLS for all endpoints
- **Field-level encryption**: PHI data protection
- **Audit logging**: All admin actions logged

### Identity Security

- **Multi-factor authentication**: Available for admin users
- **JWT tokens**: 1-hour access tokens, 30-day refresh tokens
- **Superuser auth**: Passwordless 6-digit code (no stored passwords)
- **Auth middleware**: docs and marketing CMS protected by token-based cookie middleware

### Compliance

- HIPAA compliant
- GDPR compliant
- SOC 2 aligned
- AWS BAA in place

## Ecosystem Health Monitoring

The platform includes a built-in health audit at `/api/system/ecosystem-audit` that checks:

| Category | Weight | Checks |
|---|---|---|
| Database | 20% | Connection, query latency, table existence |
| API Endpoints | 15% | Core endpoint availability |
| Security | 15% | Auth, encryption, headers |
| Deployment | 10% | Build status, domain health |
| Cost Efficiency | 5% | AI service spend |
| Voice Agent | 10% | ElevenLabs connectivity |
| Mobile App | 25% | Mobile API endpoint health |

Results are displayed in the Ecosystem Health Dashboard on both superuser portals.

- Scales from 0.5 to 16 ACUs automatically
- Pay per use, scales down during quiet periods

- Auto-scale per request
- Cold start &lt; 2 seconds
- No provisioning required
