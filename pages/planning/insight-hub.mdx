# Insight Hub — Conversational Insight as a Standalone Module

**Priority**: High — repositions the highest-engagement paid-tier feature as a discrete, sellable add-on
**Status**: Planning — scope agreed, Pass 1 (scaffolding) in progress May 2026
**Platforms**: Pro first, then mirror to University
**Builds on**: [Conversational Insight v2](/planning/conversational-insight-v2) (shipped both products)

---

## What's Changing

Conversational Insight is currently a tab inside `CompanyScopedCMS` / `UniversityScopedCMS`. Functionally it works. Commercially it doesn't — buried in a CMS settings page, it's hard to perceive as the value-added product it actually is.

This work pulls Conversational Insight out of the CMS and gives it its own standalone hub — a new Next.js app at its own subdomain, mirroring the architecture of the existing Content Hub. Same SSO handoff from admin (no separate password), same visual language, but a dedicated surface that institutions can recognise as a distinct paid module.

| | Today (shipped) | After this work |
|---|---|---|
| Surface | "Questions" tab in CMS | Standalone hub at `insight.mindmeasurepro.com` |
| Brand presence | None — invisible inside a settings tab | First-class sidebar item in admin: "Open Insight Hub" |
| Access control | Bundled with CMS access | Per-org boolean flag `conversational_insight_enabled` |
| Landing experience | Dashboard card on company overview | Full landing dashboard with KPIs, themes, quotes, HQ tips |
| Cohort comparison | Audience filter on individual questions | Dedicated cohort comparison view across any live question |
| Archive access | None — data retained but invisible | Full archive page with aggregates and CSV export |

Nothing in the underlying Conversational Insight v2 product is changing. The data model, the resolver, the per-user exposure rules, the k-anonymity floor — all unchanged. This is a repositioning and surfacing of work already shipped.

---

## Positioning Principle

Conversational Insight is not a separate product. It only works because of the engaged network of users motivated by wellbeing support — without that network there is no audience and no data. What it is, instead, is **additional value for the organisation already paying for Mind Measure**.

The hub model exists to make that value legible:

1. **It's recognisably a thing.** A buyer can see it, click into it, demo it, point at it on an invoice.
2. **It has a home for new features.** Cohort comparisons, board-pack exports, theme tracking over time, scheduled digests — all of these feel right as Insight Hub features and would feel wrong as CMS sub-tabs.
3. **It's permissionable.** (Follow-up work.) An HR analytics person who should see Insight responses but should not be able to send platform alerts gets exactly the access they need, no more.
4. **It mirrors a pattern that already works.** The Content Hub does the same thing for content/marketing. Institutions and their teams are already familiar with the pattern: click "Open Content Hub" from admin, land in a focused workspace, do the work, come back.

## Architecture

### New repo

`mind-measure-pro-insight-hub` — Next.js 16 App Router, React 19, Postgres via `pg`, lucide-react icons, inline styles with Inter + Lato fonts. Same stack as `mind-measure-pro-hub` so there's no new technology to learn.

Domain: `insight.mindmeasurepro.com`. Vercel project name: `mind-measure-pro-insight-hub`.

### Auth handoff (SSO from admin)

Identical pattern to the Content Hub:

1. Admin user clicks "Open Insight Hub" in the sidebar of `admin.mindmeasurepro.com`.
2. Admin frontend hits `POST /api/companies/[id]/insight-hub-token`, which validates that:
   - the user has company-admin access for that company, AND
   - `companies.conversational_insight_enabled = true`
3. Endpoint returns a base64-encoded JSON payload `{ email, companyId, role, ts }` and the redirect URL.
4. Browser navigates to `insight.mindmeasurepro.com/api/auth/session?token=...&name=...`.
5. Hub session endpoint sets an httpOnly cookie `insight_session` (7-day expiry) and redirects to `/dashboard`.
6. Every hub page calls `getInsightUser()` server-side; if no cookie, redirect back to admin.

No password screen on the hub. No separate identity. The hub is gated by the admin's existing auth + the per-company entitlement flag.

### Gating

One new column on `companies`:

```sql
ALTER TABLE companies
  ADD COLUMN conversational_insight_enabled BOOLEAN NOT NULL DEFAULT FALSE;
```

When `false`:
- The "Open Insight Hub" sidebar item does not render in admin.
- The token-issuing endpoint returns 403.
- Direct hits on `insight.mindmeasurepro.com` (with or without a stale cookie) get bounced to admin.

When `true` (set by Mind Measure superuser per company):
- The sidebar item renders.
- Token issuance succeeds.
- The hub serves all its pages.

Tiered pricing (basic/pro/enterprise) is deferred. A single boolean is reversible and matches what we actually know about pricing right now (zero confirmed tiers).

### Backend reuse

The Insight Hub does not introduce new business logic. Its API routes are thin proxies to the existing Conversational Insight v2 endpoints in `mind-measure-pro-core`:

| Hub route | Proxies to |
|---|---|
| `GET /api/dashboard` | `/api/companies/questions` + `/api/companies/question-responses` |
| `GET /api/questions` | `/api/companies/questions` |
| `POST /api/questions` | `/api/companies/questions` |
| `PATCH /api/questions/:id` | `/api/companies/questions` |
| `GET /api/responses/:id` | `/api/companies/question-responses?questionId=:id` |
| `GET /api/questions/:id/export.csv` | `/api/companies/questions/:id/export.csv` |
| `POST /api/cohorts/compare` | `/api/companies/question-cohort-comparison` (new — see below) |

### New backend endpoint

One new aggregation endpoint added to `mind-measure-pro-core`:

```ts
POST /api/companies/question-cohort-comparison
Body: {
  questionId: string,
  cohorts: Array<{ label: string, audience: AudienceSpec }>
}
Returns: {
  question: { id, prompt, ... },
  cohorts: Array<{
    label: string,
    cohortSize: number,
    responseCount: number,
    belowFloor: boolean,         // true if responseCount < AUDIENCE_MIN_FLOOR (5)
    themes: Array<{ theme: string, weight: number }>,
    sampleQuotes: string[]       // only populated if !belowFloor
  }>
}
```

K-anonymity floor of 5 applied **per cohort**, not just per question. A cohort with fewer than 5 responses shows "Awaiting more" instead of detail.

## Hub Pages

### `/dashboard` — Landing

Six sections, top to bottom:

1. **Top stat bar** — company name, today's date, three KPIs: active questions, total responses this period, average response rate. Prominent placement, not a footer afterthought.
2. **Live Questions panel** — list of active questions with running stats per question (Asked / Answered / Cohort), small 14-day sparkline of daily responses, click-through to the per-question deep dive.
3. **What employees are saying** — two-column treatment mirroring the Content Hub dashboard: "Most discussed themes" and "Emerging signals". Pulled from Bedrock summarisation across recent responses. K-anonymity floor of 5.
4. **Recent quotes** — anonymised sample-quote carousel, 3-4 visible cards. Each shows the question and a Bedrock-paraphrased snippet. K-anonymity already enforced in the API.
5. **From Mind Measure HQ** — content tips, suggested questions to ask this month, best-practice prompts. Mirrors the HQ suggestions block on the Content Hub.
6. **Quick actions** — "Ask a new question" primary CTA, "View archive" secondary, "Export this month" link.

### `/questions` — Question library

Port of `CustomQuestionsManager` from pro-core, rehoused in the hub with more breathing room. Author new, edit live, archive, set audience. Same underlying CRUD against `/api/companies/questions`.

### `/cohorts` — Cohort comparison

Pick a live question, build 2+ cohorts using the same profile-field-aware filter strip pattern as `CohortFilters` in pro-core, hit "Compare". Surface for each cohort:

- Cohort size (audience match against `profiles`)
- Response count (with `belowFloor` indicator if < 5)
- Theme distribution (Bedrock-extracted, bar chart per cohort)
- Sample quotes (only when `!belowFloor`)

Why this is a new component and not a port of the existing `CohortAnalysis.tsx`:

- The existing component is wired to placeholder sine/cosine data, not real data
- Its chart shapes are designed for time-series MMS comparison; this use case is distribution across cohorts for a single question
- Built fit-for-purpose with real data from day one is cleaner than dragging legacy scaffolding across

Deep link from any question's response page: `/cohorts?question=:id` preloads the question selector.

### `/archive` — Archived questions

List of archived questions with their lifetime aggregates (total asked, total answered, top themes). Per-row "Export CSV" button. Anonymous, aggregate-only — never any verbatim transcript fragments.

### `/responses/[id]` — Per-question deep dive

For a single live question: response timeline (daily chart), audience reach breakdown, paraphrased sample quotes, "Compare cohorts" CTA that routes to `/cohorts?question=:id`.

### `/login` — Fallback bounce

If anyone hits `insight.mindmeasurepro.com` without a valid cookie (and isn't coming through the token endpoint), bounce to `https://admin.mindmeasurepro.com` with a flash message.

## What's Removed from Admin

Once the Insight Hub is live and verified:

- The "Questions" tab in `CompanyScopedCMS.tsx` is removed.
- The "Conversational Insight" card on the company Overview dashboard is **kept** — it still surfaces running stats in the main admin, but the configure CTA now opens the hub.
- Existing `/api/companies/questions` and `/api/companies/question-responses` endpoints stay exactly as-is — the hub calls them, the mobile resolver calls them.

The removal is a separate commit, separate deploy from the hub launch — don't fly the cutover without a net.

## Build Order (Five Passes)

Sized to allow validation at each step rather than landing as a single 3-day PR.

### Pass 1 — Scaffolding (~½ day)

- Create `mind-measure-pro-insight-hub` repo
- Copy HubShell pattern, brand tokens, layout from pro-hub
- Port auth handoff: `getInsightUser()`, `/api/auth/session`, `/api/auth/logout`
- Stub all routes (`/dashboard`, `/questions`, `/cohorts`, `/archive`, `/responses/[id]`, `/login`) with placeholder content
- Add the gating migration + runner endpoint in pro-core
- Add `POST /api/companies/[id]/insight-hub-token` to pro-core
- Add the "Open Insight Hub" sidebar item in admin, gated on the flag

**Validation**: Mind Measure superuser sets the flag for one company → company admin sees the sidebar item → clicks it → lands on the (placeholder) hub dashboard.

### Pass 2 — Questions surface (~½ day)

- Port `CustomQuestionsManager` UI into `/questions`
- Wire to `/api/companies/questions` via hub proxy
- Confirm audience picker still works (port `AudiencePicker` if needed)

**Validation**: Hub admin creates, edits, archives questions; changes appear in admin's existing dashboard card unchanged.

**Shipped**: Questions list (live / drafts / archived), compose dialog (text, internal note, priority, duration), publish / pause / archive / restore actions. Auth bridge: hub mints a short-lived `iss: 'mindmeasure-platform'` JWT signed with `SUPERUSER_JWT_SECRET`, carries only the caller's email; pro-core's `resolveCompanyAdmin` does the actual authorisation against `company_authorized_users`. Audience defaults to "Everyone" for new hub-authored questions; existing cohort-targeted questions are preserved on edit because PATCH omits the `audience` field unless explicitly set.

### Pass 2.1 — Audience picker (~½ day)

Defers the full `AudiencePicker` port from Pass 2 because the picker pulls in a non-trivial dependency tree (shadcn primitives, `useReachEstimate` hook, `companyProfileFields` config, `parseProfileFieldsConfig`) that doesn't yet exist in the hub.

- Port `AudiencePicker` and its dependencies to the hub
- Wire the reach-estimate proxy (`/api/questions-reach-estimate` -> pro-core `/api/companies/questions-reach-estimate`)
- Replace the "Audience: Everyone" hint in the compose dialog with the real picker

**Validation**: Hub admin creates a new question targeted at a specific cohort (e.g. Tax department only); resolver only exposes it to employees matching that cohort.

**Shipped**: Picker rebuilt for the hub with native HTML + Tailwind (no shadcn primitive port needed). Company row (divisions / departments / offices / profile_fields / total_employees) is server-rendered alongside the question list in a single SSR pass via `src/lib/company.ts`, so cascading dropdowns are populated on first paint. Reach estimate hits the new `/api/questions-reach-estimate` hub proxy, which forwards to pro-core's `/api/companies/questions-reach-estimate` via the bridge JWT and applies the k-anonymity floor of 5. POST and PATCH from the hub now always include the audience field so picker edits round-trip correctly.

### Pass 3 — Landing dashboard (~1 day)

- Build the six-section dashboard
- KPIs, live questions panel, themes, quotes, HQ tips, quick actions
- This is the bit that defines whether it feels like a product

**Validation**: Open the hub with a populated dataset (PwC) and see a magazine-style landing page comparable to the Content Hub.

**Shipped**: Four of the six planned sections — top stat bar (active questions, responses in window, average response rate), Live Questions panel (expandable: sentiment bar + Bedrock summary + anonymised paraphrased samples + audience), Recent Quotes carousel (aggregated samples across questions), Quick Actions (links into Questions / Cohort Compare / Archive). Window selector (30 / 90 / 365 days) drives the entire dashboard. K-anonymity floor of 5 enforced server-side by pro-core's `/api/companies/question-responses` — no hub-side reimplementation.

### Pass 3.1 — Sparklines, themes, HQ tips (~½–1 day)

Carves out the two remaining sections of the spec plus per-question 14-day sparklines that needed pro-core endpoint work.

**Shipped:**

- **14-day response sparklines** on every Live Question card. New `GET /api/companies/question-responses-timeline?companyId=&questionId=&days=` endpoint in pro-core returns one row per day in the window (uses `generate_series` so zero-response days still appear — no client-side date backfilling). Window-decoupled from the main dashboard window: sparklines are always 14d because they exist to answer "is this question alive RIGHT NOW", not to mirror the larger aggregate. Fetched in parallel client-side after mount so the dashboard's first paint isn't blocked on N round-trips. Rendered with a native SVG component (`src/components/charts/Sparkline.tsx`) — no chart library dependency.
- **"What employees are saying"** section: cross-question themes. New `GET /api/companies/cross-question-themes?companyId=&days=` endpoint sends every active question's last-30d responses to Bedrock (Claude 3 Haiku) in a single pass grouped by question, returning two buckets: `discussed` (3–5 dominant themes that span multiple questions or are heavily mentioned in any single one) and `emerging` (2–4 distinct low-frequency signals worth watching). Per-question response cap (25) + corpus cap (200) with round-robin sampling so a chatty question can't monopolise the prompt. K-anonymity applied at the **corpus level** (5 total responses across all active questions, not per question) — the whole point of cross-question aggregation is to surface signal that wouldn't clear a per-question floor on its own. Theme-extraction failures return empty buckets non-fatally; the dashboard renders a graceful "AI summary unavailable right now" state.
- **From Mind Measure HQ** tips block: three hand-curated tip cards (keep questions live for at least a fortnight, cohort-target sensitive topics, use cohort comparison after a change). Static content for v1 — considered driving from the marketing CMS but tips are stable product guidance, not weekly content. If/when this becomes a regular touchpoint we'll move it.
- **Export last Nd** quick action: replaces the previous "Manage questions" action card (which was duplicative — it routed to the same place as "Ask a new question"). Generates a client-side CSV bundle of every question's aggregates for the current dashboard window. Aggregates only — no verbatim quotes, paraphrased samples excluded for safety. CSV-injection-guarded (cells starting with `=+-@` get a leading apostrophe), RFC-4180-compliant.

Supporting infrastructure: `src/lib/csv.ts` was extracted with shared `escapeCsv` / `rowsToCsv` / `downloadCsv` helpers so the dashboard export and the existing archive per-row export share one implementation.

### `/responses/[id]` — Per-question deep dive (shipped)

Per-question deep dive page replaces the Pass 1 placeholder. Server-renders the question aggregate (last 30d) and 90-day daily timeline in parallel via two upstream calls, so first paint arrives with everything populated.

**Page layout:**

- Back-to-dashboard link.
- **Header**: question prompt + status pill + audience summary + active-from/until dates + "Compare cohorts" CTA (deep-links into `/cohorts?question=:id`).
- **KPI strip**: asked / answered / response rate / cohort.
- **Response timeline**: 90-day sparkline (`<Sparkline>` component shared with the dashboard) with peak day, daily average, and active-day count callouts.
- **What employees told us**: sentiment bar + Bedrock summary + paraphrased samples. Mirrors the dashboard's expanded panel but bigger and permalinked.
- **Audience reach**: visualised cohort breakdown with k-anon-clamped reach number.

Below the k-anonymity floor of 5, detail sections collapse to the same "Awaiting more responses" lock state the dashboard uses, so narrow-cohort questions stay opaque even when permalinked. Stale bookmarks for deleted/wrong-company question IDs surface a `notFound()` rather than a 500.

The dashboard's Live Question cards each include a "View full details →" footer link to the deep dive plus a "Compare cohorts" shortcut for parity with the archive page.

### Pass 4 — Cohort comparison, archive, export (~1 day)

- Build `/cohorts` and the new `/api/companies/question-cohort-comparison` endpoint
- Build `/archive` with CSV export
- Remove the Questions tab from `CompanyScopedCMS.tsx` (separate commit, separate deploy)

**Validation**: Pick the most-answered PwC question, build "Tax vs Audit" cohorts, see distinct theme distributions. Export the archive as CSV.

**Shipped (endpoint + UI)**:

- New `POST /api/companies/question-cohort-comparison` in pro-core. Takes `{ companyId, questionId, cohorts: [{ label, audience }], days? }`, returns per-cohort `{ label, cohortSize, responseCount, belowFloor, themes: [{theme, weight}], sampleQuotes }`. Lifetime-by-default, optional `days` window override capped at 365. K-anonymity floor of 5 applied PER COHORT (not just per question). Cohort sizes for filtered audiences are clamped to the floor so a narrow filter can't reveal a small headcount. Bedrock theme extraction (Claude 3 Haiku) capped at 25 samples per cohort to bound token spend; failures degrade gracefully to empty themes + visible sample quotes.
- Hub `/cohorts` UI: question selector (preselects from `?question=:id` deep link), 2–6 dynamic cohort builders each with the full AudiencePicker + label input, Compare button, per-cohort result cards with horizontal theme bars, sample quotes and below-floor lock state. Native HTML + Tailwind throughout — no chart libraries added.
- Hub `/archive` UI: list of archived questions with asked / answered / response rate / cohort running stats. Expandable row reveals sentiment bar + Bedrock summary + paraphrased samples. Per-row `CSV` button generates an anonymised single-row CSV client-side (aggregate counts, sentiment buckets, summary, samples — never verbatim transcripts). Each row also has a `Compare` button that deep-links into `/cohorts?question=:id`. CSV-injection guard on cells starting with `=+-@`.

**Deferred** (separate commit after hub is verified): remove the Questions tab from `CompanyScopedCMS.tsx`. The hub-only flow needs a couple of days of smoke testing first so we don't strand admins mid-edit if there's a bug in the hub-side CRUD.

Affects all companies regardless of `conversational_insight_enabled` — the spec was always full removal. The Overview's empty-state Conversational Insight card now deep-links to `?section=insight` (the Insight Hub launch card) instead of `?section=questions`. The `CustomQuestionsManager.tsx` source file is left on disk as dead code; a housekeeping commit can delete it once we're confident nothing else references the symbol.

## Passwordless Front Door (shipped)

The hub now has its own sign-in page at `insight.mindmeasurepro.com/login`. Company admins on `company_authorized_users` can land direct at the hub URL — typed, bookmarked, or from a welcome email — without going through `admin.mindmeasurepro.com` first.

### Why ship this now

Better demos (one URL to send a buyer's HR analytics team), and a cleaner mental model for the people who think of Insight Hub as "the conversational insight product" rather than "a thing inside the admin tool". Mind Measure staff still come through the admin SSO sidebar; the front door is specifically for institutional admins.

### How it works

The hub doesn't have its own auth stack. It proxies onto pro-core's existing `POST /api/auth/send-access-code` and `POST /api/auth/verify-access-code` (the same endpoints `SuperuserLogin` uses), which already know how to:

- Resolve an email against `core_users` AND `company_authorized_users`.
- Mint a JWT challenge token tied to a hashed 6-digit code (10-minute expiry, signed with `SUPERUSER_JWT_SECRET`).
- Send the code via SES from `noreply@mindmeasurepro.com`.
- Rate-limit per IP via `@mindmeasure/api-lib/rate-limiter`.

Reuse over reinvention. The only hub-specific work is on the wrapper:

| Hub route | Purpose |
|---|---|
| `POST /api/auth/email-login/request` | Forwards `{ email }` to pro-core send-access-code. Passes `x-forwarded-for` through so the rate limiter keys per-user, not per-hub-instance. Returns `{ challengeToken }`. |
| `POST /api/auth/email-login/verify` | Forwards `{ email, code, challengeToken }` to pro-core. On success, runs two extra gates before issuing a session cookie. |
| `GET /login` | Server-rendered page that bypasses the form if `getInsightUser()` returns a user (already signed in → straight to `/dashboard`). Otherwise renders the two-step client form. |
| `GET /api/auth/logout` | (unchanged) Clears the `insight_session` cookie and bounces to `admin.mindmeasurepro.com`. |

### Verify-side gates

After pro-core verifies the code, the hub applies two further checks before promoting the result into a session cookie:

1. **No `companyId` on the verified user** → 403 with "Mind Measure staff: sign in via admin and use the *Open Insight Hub* sidebar link instead." Staff coming via `core_users` have platform-wide permissions, not a single-company scope, and we don't ship a company picker for v1.
2. **`conversational_insight_enabled = false`** on the company → 403 with "Insight Hub isn't enabled for {company name}. Contact your Mind Measure account manager." Catches the case where an admin's email is on the access list but their org hasn't bought the module. Without this check the user would log in successfully, then `getInsightUser()` would bounce them silently on the next request, which is a confusing UX.

Only if both gates pass do we mint a `base64(JSON)` session token via `createSessionToken(email, companyId, role, companyName)` and set it as `insight_session` with `httpOnly + secure + sameSite=Lax + maxAge=24h`. From the cookie's perspective, the front-door flow and the SSO handoff are indistinguishable — `getInsightUser()` reads the same shape either way and re-validates the entitlement flag on every request.

### Client UI

`/login` mirrors `SuperuserLogin` from pro-core for platform consistency:

- Full-bleed backdrop image (`public/fabien-bellanger-WyfXOHgI49s-unsplash.jpg`) with a darken-gradient overlay so the white form card pops at any breakpoint.
- Left at `lg+`: backdrop-blur branding panel — "Conversational insight, on demand" headline, copy describing the cohort/anonymity story, security note about passwordless sign-in.
- Right: form card. Step 1 is an email input with the access-list explainer; step 2 is six separate digit inputs that auto-advance, accept clipboard paste, and auto-submit on the sixth digit. Resend code, change email, and a footer link routing Mind Measure staff to admin.

### Follow-ups for this surface

- **Image optimisation**: the backdrop is 8MB on disk. Run it through a build-time compress (or swap to next/image with a smaller asset) before any high-traffic demo.
- **Welcome email / new-admin onboarding**: when a company admin is added via the admin app's user-management UI, the invite email currently sends them to `admin.mindmeasurepro.com`. For Insight-Hub-only admins it could send them straight to `insight.mindmeasurepro.com/login` instead.
- **University parity**: the mirror to `mind-measure-uni-insight-hub` will need the same passwordless flow against uni-core's `university_authorized_users`. Same architecture, different table.
