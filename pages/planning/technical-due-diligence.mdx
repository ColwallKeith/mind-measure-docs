## 2. Intellectual Property: Multimodal Fusion Engine

The platform's core differentiator is a **3-modality wellbeing assessment engine** that analyses check-in conversations across:

- **Text analysis (70% weight)**: AWS Bedrock (Claude 3.7 Sonnet) extracts mood scores, PHQ-2/GAD-2 clinical indicators, risk markers, sentiment, key themes, positives, and negatives from conversation transcripts.
- **Audio analysis (15% weight)**: Client-side Web Audio API extracts speech patterns â€” energy levels, speaking rate, vocal stress indicators, silence ratios.
- **Visual analysis (15% weight)**: AWS Rekognition analyses facial expressions during check-ins â€” emotion classification, engagement indicators.

The fusion engine includes **dynamic fallback**: if one modality fails (e.g., camera permission denied), weights automatically redistribute across available modalities. The system degrades gracefully rather than failing.

Session processing is **asynchronous**: completed check-ins are queued via SQS and processed by Lambda workers, preventing user-facing latency from downstream processing.

This architecture is non-trivial to replicate. The combination of real-time voice AI, multimodal fusion with fallback, and clinical indicator extraction (PHQ-2, GAD-2) in a single check-in flow represents genuine technical IP.

## 3. Security Architecture

### 3.1 Authentication

**Primary authentication**: AWS Cognito with RS256 JWT tokens. All API routes verify JWT signatures against the Cognito JWKS endpoint â€” tokens are cryptographically verified, not merely decoded.

**Superuser authentication**: Separate flow using time-limited 6-digit codes sent to verified @mindmeasure.co.uk email addresses. Codes are HMAC-hashed (not stored in plaintext). Challenge tokens expire after use.

**MFA**: AWS Cognito MFA with TOTP (Google Authenticator) and SMS support. MFA setup, challenge, and verification endpoints exist at `/api/auth/mfa-setup`, `/api/auth/mfa-challenge`, `/api/auth/mfa-verify`.

### 3.2 Encryption

| Layer | Method | Standard |
|-------|--------|----------|
| Data at rest (database) | AES-256 | AWS KMS, annual key rotation |
| Data at rest (Cognito) | AES-256 | AWS KMS |
| Data in transit (browser â†’ Vercel) | TLS 1.3 | HTTPS enforced |
| Data in transit (Vercel â†’ Aurora) | TLS 1.2+ | SSL required (`sslmode=require`) |
| Data in transit (email) | TLS 1.2 | STARTTLS via SES |
| Field-level PHI encryption | AES-256-GCM | Client-side, authenticated encryption with AAD |

**PHI field-level encryption** (`PHIEncryption.ts`) encrypts sensitive fields individually: first name, last name, email, phone number, date of birth, medical record numbers, conversation transcripts, assessment results, medical notes, diagnostic data. This is defence-in-depth â€” even if the database is compromised, PHI fields remain encrypted.

### 3.3 Access Control (RBAC)

| Role | Scope | Capabilities |
|------|-------|-------------|
| Student / Employee | Own data only | Check-ins, view own history, export own data |
| University Admin / Company Admin | Scoped to institution | View aggregated analytics, manage users, generate reports |
| Superuser | Platform-wide | All admin functions, system configuration, ecosystem audit |

RBAC is enforced via `requireRole()` middleware on API routes. Data scoping ensures institution admins can only access data within their organisation.

### 3.4 Input Validation & Injection Prevention

- **Parameterised SQL**: Every database query uses `$1, $2` parameterisation â€” zero SQL injection surface.
- **Table allowlists**: Database endpoints only accept queries against pre-approved tables.
- **Identifier validation**: Column and table names validated against `/^[a-zA-Z_][a-zA-Z0-9_]*$/`.
- **Frame array caps**: Visual analysis endpoints cap input arrays to prevent abuse.
- **Transcript length limits**: Text analysis endpoints enforce maximum input sizes.

### 3.5 CORS Configuration

CORS is **whitelist-based** (not wildcard). Each project's `api/_lib/cors-config.ts` defines an explicit origin allowlist:
- Production domains (admin, mobile, hub for each platform)
- Capacitor/Ionic localhost (for mobile apps)
- Development localhost (only in dev mode)

Preflight requests are handled correctly. Credentials are allowed only for whitelisted origins.

### 3.6 Security Headers

Content Security Policy, X-Frame-Options, X-Content-Type-Options, Referrer-Policy, and XSS protection headers are set via Next.js middleware:

- `Content-Security-Policy`: Restricts script sources, prevents frame embedding (`frame-ancestors 'none'`)
- `X-Frame-Options: DENY`
- `X-Content-Type-Options: nosniff`
- `Referrer-Policy: no-referrer`

### 3.7 Audit Logging

Two-tier audit logging system:

**API-level** (`api/_lib/audit-logger.ts`): Fire-and-forget async logging with IP extraction, risk classification, convenience functions for auth events, data access, and data modification. HIPAA Â§164.308 compliant.

**Service-level** (`services/security/AuditLogger.ts`): Database-backed logging with compliance flagging (HIPAA_RELEVANT, GDPR_RELEVANT), risk level calculation, and action type classification (LOGIN_SUCCESS, PHI_VIEW, PHI_EXPORT, etc.).

Audit logs are stored in an **immutable table** protected by database triggers â€” they cannot be modified or deleted. Retention period: 7 years.

### 3.8 Security Database Infrastructure

8 security tables and 5 security views are defined in migration `010_create_security_infrastructure.sql`:

**Tables**: `audit_logs`, `security_incidents`, `encryption_status`, `compliance_checks`, `data_subject_requests`, `incident_response_team`, `security_policies`, `security_metrics`

**Views**: `v_recent_phi_access` (24h PHI access log), `v_mfa_enrollment` (MFA by role), `v_open_incidents` (open incidents by severity), `v_compliance_summary`, `v_data_subject_requests` (GDPR requests with SLA tracking)

### 3.9 Vulnerability Scanning

`VulnerabilityService.ts` provides a framework for dependency scanning (npm audit integration), compliance scanning (HIPAA, GDPR, SOC2, ISO 27001, NIST), CVE/CVSS scoring, and remediation tracking.

## 10. Assessed Risk Areas

### 10.1 Vendor Dependency â€” ElevenLabs

The voice check-in feature (core product differentiator) depends on ElevenLabs Conversational AI. ~35 files touch this integration. A voice provider abstraction layer is planned on the roadmap to mitigate lock-in risk, with NVIDIA PersonaPlex identified as a potential self-hosted alternative at scale.

**Mitigation**: Provider abstraction is planned. The conversation data flow (transcripts, analysis) is platform-agnostic â€” only the real-time voice layer is ElevenLabs-specific.

### 10.2 Application-Level Rate Limiting

API-level rate limiting middleware is not explicitly implemented. The platform relies on:
- Vercel's built-in DDoS protection
- Database-level rate limiting for specific operations (e.g., buddy check-in requests)
- Input validation caps on array sizes and transcript lengths

**Mitigation**: Vercel Pro provides enterprise DDoS protection. Database-level rate limiting exists for sensitive operations. Application-level middleware is a recommended enhancement.

### 10.3 Error Tracking

**Sentry** is integrated across all 4 active applications (both admin dashboards and both mobile apps):
- **Client-side**: `@sentry/react` with browser tracing, session replay on error (text masked for HIPAA), and `ErrorBoundary` wrapping at the app root
- **Server-side**: `@sentry/node` on both admin API backends, with 5xx errors automatically reported via the centralised error handler
- **Environment tagging**: Events are tagged by platform (university/pro) and app type (admin-dashboard/mobile/api) for filtering
- **Privacy**: Session replay masks all text and blocks all media. Sentry is disabled in localhost environments.

Additional error visibility comes from Vercel deployment logs, AWS CloudWatch/CloudTrail, the ecosystem health audit, and immutable audit logging.

### 10.4 Bus Factor

The platform has been built primarily by a single developer. This is mitigated by:
- Comprehensive documentation site (docs.mindmeasure.co.uk)
- Consistent patterns across all projects (same stack, same structure)
- 86+ automated tests with meaningful assertions
- CI/CD pipelines that enforce quality standards
- TypeScript at maximum strictness (self-documenting code)
